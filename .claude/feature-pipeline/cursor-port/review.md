# Review: cursor-port

1. **[HIGH | confidence: HIGH]** `${CURSOR_PROJECT_DIR}` is wrong for hook command paths
   The plan's `hooks/hooks.json` entry for Cursor uses `bun ${CURSOR_PROJECT_DIR}/scripts/hooks/capture-session.ts`. But `CURSOR_PROJECT_DIR` resolves to the **user's project root** (where they invoked Cursor), not the plugin install directory. The plugin lives in Cursor's plugin cache (location not stable across versions). The `cursor/plugin-template` repo (`plugins/starter-advanced/hooks/hooks.json`) uses RELATIVE paths like `"./scripts/format-code.sh"` which Cursor resolves relative to the plugin root. The plan must change all Cursor hook entries to `"./scripts/hooks/capture-session.ts"` form, not `${CURSOR_PROJECT_DIR}/...`. Otherwise both Cursor hooks will fail-to-start because the path resolves to a non-existent location in the user's project.

2. **[HIGH | confidence: HIGH]** SKILL.md bodies' `${CLAUDE_PLUGIN_ROOT}` will be empty on Cursor
   The 9 SKILL.md files contain shell commands like `bun ${CLAUDE_PLUGIN_ROOT}/scripts/cli/advance.ts set ...`. `CLAUDE_PLUGIN_ROOT` is a Claude-Code-specific env var. Cursor's hooks docs list `CURSOR_PROJECT_DIR`, `CURSOR_VERSION`, `CURSOR_USER_EMAIL`, `CURSOR_TRANSCRIPT_PATH`, `CURSOR_CODE_REMOTE`, and `CLAUDE_PROJECT_DIR` (alias) — but **NO** plugin-root env var. When the model runs the bash commands quoted in the SKILL.md body on Cursor, `${CLAUDE_PLUGIN_ROOT}` will expand to an empty string and the command becomes `bun /scripts/cli/advance.ts ...` — fails. The plan does not address this. Options: (a) wrap `advance.ts` in an installed CLI shim (Cursor doesn't expose a PATH-injection mechanism); (b) hardcode an absolute path that's resolvable on the user's machine via npx/bunx; (c) have the orchestrator skill compute the plugin root once at session start and store it in `DP_PLUGIN_ROOT` (and the sessionStart hook can in fact return `env: {DP_PLUGIN_ROOT: ...}` if it can determine its own dir via `import.meta.dir`). Option (c) is the cleanest — sessionStart hook propagates `DP_PLUGIN_ROOT` alongside `DP_SESSION_ID`, and SKILL.md bodies switch from `${CLAUDE_PLUGIN_ROOT}` to `${DP_PLUGIN_ROOT}`. This breaks Claude Code unless we also have the Claude Code SessionStart hook export `DP_PLUGIN_ROOT`. Plan needs this explicit decision and the file-by-file updates that fall out.

3. **[HIGH | confidence: HIGH]** Plan reverses the proposal's approved skill-renaming decision without flagging it
   The approved proposal section 2 explicitly stated: "Rename every skill in frontmatter from `name: investigation` to `name: dp-investigation` (and equivalent for the other 8)." The plan's Approach section now says "Skill directory names stay as today" and "The SKILL.md frontmatter `name:` field stays at the short form (`investigation`, `plan`, …)." This is a direct reversal. The rationale (avoid breaking Claude Code's `/dp:investigation`) is plausible, but per the user's strict "no silent deviation from approved plan" rule, the plan must either (a) honor the proposal and prefix the names (and accept that Claude Code shows `/dp:dp-investigation`), OR (b) explicitly call out the reversal in the Context section, with rationale, and have the user re-approve. As written, the plan-improve-apply step would silently propagate the reversal into implementation. Flag at HIGH because it's a discipline failure, not just a technical issue.

4. **[MEDIUM | confidence: MEDIUM]** Dual-format `hooks/hooks.json` coexistence is conjecture, not verified
   The plan asserts: "each platform reads only its own key naming, silently ignores unknown event names." Cursor docs and Claude Code docs neither confirm nor deny this. The two formats differ structurally: Claude Code uses `hooks.<PascalCase>.[i].hooks.[j].{type,command}` (deep nesting); Cursor uses `hooks.<camelCase>.[i].{command,loop_limit,matcher?}` (flat). If either parser is strict-schema (validates known events only and rejects on unknown keys), the dual file breaks both installs. Fallback: split into two hook files, each scoped under its manifest. Claude Code's hook config can in fact be carried inline in skill frontmatter or a per-plugin `hooks/hooks.json` referenced from the manifest. Cursor's `.cursor-plugin/plugin.json` accepts an explicit `hooks: "./path/to/hooks.json"` field. So a two-file approach (`.cursor-plugin-hooks.json` for Cursor, current `hooks/hooks.json` for Claude Code) is feasible. The plan should either (a) include a verification step that explicitly checks one-file dual coexistence loads cleanly on BOTH platforms before declaring done, OR (b) preemptively use two files. Without verification, the risk is shipping broken installs.

5. **[MEDIUM | confidence: HIGH]** `disable-model-invocation` policy not addressed
   The SKILL.md files currently have NO `disable-model-invocation` field (it was removed earlier in the project's history because it broke Claude Code's `Skill` tool). On Cursor, the model can description-match any skill and auto-invoke it. The 9 step skills have descriptions like "Use when an active dev-pipeline run is at the plan step" — without `disable-model-invocation: true`, on Cursor the model could trigger `plan` (or any step) outside of an active pipeline run based purely on description match. This is exactly the corruption scenario the field exists to prevent. The auto-prompt chaining doesn't need this field because `followup_message` containing `/<skill>` invokes via slash-prefix regardless of the flag. The plan should decide: re-add `disable-model-invocation: true` to all 9 step skills (works on Cursor; on Claude Code the Skill tool would break again, but since we have the auto-prompt fallback, that's now acceptable). OR leave it off and accept the auto-trigger risk on Cursor. Either is defensible but the plan must pick one and document it.

6. **[MEDIUM | confidence: HIGH]** Plan claims Cursor's `commands/` directory is unused; the official template ships one
   Plan: "Cursor's commands/ slash-command primitive is documented but unused by every official Cursor plugin, and isn't surfaced as a user-facing `/`-menu entry." Verification against `cursor/plugin-template` shows `plugins/starter-advanced/commands/` exists. The plan's claim is overstated. Two consequences: (a) the orchestrator could potentially be a `commands/dev-pipeline.md` on Cursor too (avoiding the `dp-dev-pipeline` skill duplication); (b) the rationale for using a skill needs to be re-justified. Suggest the plan either (i) tests whether Cursor surfaces `commands/*.md` in the `/`-menu, OR (ii) softens the claim to "commands/ is supported but not yet ubiquitous; we use a skill for consistency with the auto-prompt chain (which only triggers slash-prefixed skills, not commands)."

7. **[MEDIUM | confidence: HIGH]** `dp:improve` meta-skill changes not enumerated
   `skills/improve/SKILL.md` is the meta-skill that bumps version in `.claude-plugin/plugin.json` after a self-modification. The plan adds `.cursor-plugin/plugin.json` AND `.cursor-plugin/marketplace.json`, both of which carry their own `version` field. Now we have 4 versioned files that must stay in lockstep (Claude plugin.json, Claude marketplace.json's two version fields, Cursor plugin.json, Cursor marketplace.json's two version fields). Without an update to `skills/improve/SKILL.md`, the next `/dp:improve` invocation bumps only the Claude side, creating version drift. The plan must add a file-by-file entry for `skills/improve/SKILL.md` describing what changes (read all 4 manifests, bump all `version` occurrences in lockstep, fail loudly if any goes out of sync).

8. **[MEDIUM | confidence: MEDIUM]** Flat skill-name collision risk on Cursor not concretely assessed
   Plan section "Skill renames are deliberately minimal" cites the risk of `/investigation`, `/plan` colliding with other installed Cursor plugins. The plan does not check (a) whether any of the 9 official Cursor plugins (`cursor/plugins` repo: continual-learning, cursor-team-kit, create-plugin, agent-compatibility, cli-for-agent, pr-review-canvas, docs-canvas, cursor-sdk, orchestrate) ship a skill named `plan`, `investigation`, `codereview`, etc.; (b) whether popular community plugins do. From sample inspection earlier in the run: the official Cursor plugins don't seem to use those names, but `compound-engineering` is a known popular plugin and ships `/ce-plan`, `/ce-code-review` (prefixed). So there's NO known collision in the immediate ecosystem. The plan should add a one-line verification: "Confirmed by inspecting `cursor/plugins` official repo that no top-9 plugin ships a skill matching `investigation`, `plan`, `plan-proposal`, `plan-improve`, `plan-improve-apply`, `plan-wrapup`, `implementation`, `codereview`, `improve` as of plan date." Without that note, the risk feels hand-waved.

9. **[LOW | confidence: HIGH]** Cursor `hooks.json` command path examples in the plan use unverified syntax
   Beyond issue #1, the plan's example syntax for Cursor's `stop` entry is partly invented. The plan reads `[{command: "bun ./scripts/hooks/...", loop_limit: null}]`. The `cursor/plugin-template` example uses `{command: "./scripts/...", matcher: "..."}` — no `bun` prefix because the script is a `.sh` shebanged shell script. If we want Cursor to invoke a Bun TypeScript file, the command needs to be `bun ./scripts/hooks/capture-session.ts` literally — which works because Bun is on PATH. Confirm the plan's path is right after fixing issue #1, but worth flagging that no template-provided example shows a Bun-style command.

11. **[LOW | confidence: MEDIUM]** Verification step for cross-platform mixed-session isolation is incomplete

Add a verification step: "Open the same project in both Claude Code and Cursor simultaneously, start a different pipeline in each, confirm neither session's Stop hook triggers on the other's active run."
